# Abans Kiosk E-Catalog — Project Reference

Status: **requirements captured, no visual design started yet.** This file is the source of truth for scope, device constraints, and design tokens. Design work begins in a later phase.

## 1. Project Overview

A prototype for a digital product e-catalog for **Abans**, designed for an in-store **kiosk device**. The app showcases the Abans product catalog and lets a user request a spec sheet for a selected product. The spec is delivered to the user by email.

Core user flow (proposed):
1. User browses/searches the product catalog.
2. User selects a product to view details.
3. User requests the product spec.
4. Email is collected — likely early in the flow (e.g. at entry or before browsing) rather than only at the point of request, so it isn't a blocker at the moment of conversion. This is a working idea, not finalized — revisit during flow design.
5. Spec sheet is emailed to the user.

Open questions to resolve during design/flow phase:
- Exact screen where email is captured (kiosk entry/attract screen vs. pre-spec-request step).
- What "spec" contains (PDF spec sheet, structured data, images) and how it's generated.
- Idle/attract state and session timeout/reset behavior (kiosk best practice).
- Search vs. category-browse as primary navigation model.
- On-screen keyboard behavior for email entry on a touch kiosk.

## 2. Target Device & Display

- Device type: **Kiosk** (touch, likely no hardware keyboard/mouse).
- Display size: **43"**.
- Orientation: **Portrait**.
- Aspect ratio: **9:16**.
- Resolution: **1920 × 1080** (H × W as specified — i.e. 1080 wide × 1920 tall in portrait).
- Design at native resolution: **1080px width × 1920px height** canvas/viewport for all screens.
- Touch-first interactions: large tap targets, no hover-dependent UI, account for viewing distance on a large-format display.

### 4.5 Typography

- Font: **SF Pro**.

**Base type scale (proposed).** Derived from the actual display, not a generic web scale: at 43" diagonal / 1080×1920px, this panel is ~51 PPI — roughly half the pixel density of a typical laptop or tablet screen (~100–110 PPI). That means each CSS px covers more physical space here, so the scale below sits well above typical web sizing even though the numbers look large. Assumes 1 CSS px = 1 device px (no OS/browser display scaling) — verify against the real hardware once it's available, since a scaled OS config would change these physical sizes.

| Token | px | Use | Cap-height (physical) |
|---|---|---|---|
| Caption / Meta | 24 | Tags, timestamps, helper text | ~8.5 mm |
| Body Small | 28 | Secondary copy, list rows | ~9.9 mm |
| Body Base | 32 | Default paragraph / description text | ~11.3 mm |
| Body 
 / Button label | 36 | Primary buttons, form labels | ~12.7 mm |
| Card Title (H4) | 44 | Product card names in grid/list | ~15.6 mm |
| Section Title (H3) | 56 | Category headers, section dividers | ~19.8 mm |
| Screen Title (H2) | 72 | Page/screen headings | ~25.5 mm |
| Product Name (H1) | 88–104 | Product detail hero title | ~31–37 mm |
| Display / Attract | 128+ | Idle/attract screen, large callouts | 45 mm+ |

Notes:
- Smallest step (24px / ~8.5mm cap-height) is already well above the ADA minimum for interactive kiosk text (3/16in ≈ 4.8mm), leaving margin for viewing distance and glare on a public-facing display.
- Line-height: ~1.4–1.5 for body copy, ~1.15–1.25 for headings (tighter as size increases) — fine-tune per weight during build.
- SF Pro weights available: Regular (body), Medium (labels/emphasis), Semibold (headings/buttons), Bold (display/hero) — pair weight with scale step rather than introducing more sizes.
- This is a starting scale, not final — validate legibility and touch-label sizing against physical hardware before locking in.

### 4.7 Motion / Animation

- **Library: [Motion](https://motion.dev)** (formerly Framer Motion). Install as `motion`, import from `motion/react` (not the legacy `framer-motion` package). Confirmed compatible with React 19 and Next.js 16 App Router as of March 2026 — components using it need the `"use client"` directive.
- **Loading animations** are part of the prototype — used for states like content/data loading, spec-request submission, and email send confirmation. Use `AnimatePresence` for enter/exit and loading-state transitions.
- **Scroll animations** are part of the prototype — used to add polish/engagement as the user scrolls through the catalog and product detail screens (e.g. reveal/fade-in of cards, staggered entrances). Use `whileInView` / `useScroll` for scroll-driven reveals.
- Motion should feel smooth and premium on the large-format kiosk display but not slow down or block touch interactions — kiosk users expect responsiveness.
- Specific durations, easing curves, and per-component animation choices are not yet decided — to be defined during the design/build phase.

### 4.8 Primary CTA Button — Attract & Tap Motion

Approved reference behavior for primary call-to-action buttons (e.g. "Request spec"), designed to draw attention on an idle kiosk screen and give unambiguous touch feedback:

- **Fill**: solid brand color (`#781E7D`, purple/500) — flat, no gradient.
- **Idle "flash sweep"**: a sharp-edged, skewed light bar (~14% of button width, `rgba(255,255,255,0.55)`, no blur/feather) sweeps left → right across the button on a loop (~3s cycle: fast sweep in the first ~0.5s, then a pause before repeating). Clipped to the button's pill shape via `overflow: hidden`.
- **Tap feedback**: on press, the button scales **up** to ~1.08x (not down — deliberate, for visibility on a large display) using a springy easing (`cubic-bezier(.34,1.56,.64,1)`, ~150ms), the fill brightens to purple/400 (`#A062A4`), and a brief full-button white flash (`rgba(255,255,255,0.55)` fading out over ~250ms) reinforces the tap. Returns to idle state on release.
- No glow/blur/box-shadow effects — kept sharp and flat, consistent with the "sharp" brand direction (distinct from soft/neon glow treatments, which were explicitly rejected).
- Prototyped and approved as an interactive HTML demo; implement in Motion (`motion/react`) using `whileTap` for the press state and a looping `animate` (or CSS keyframes) for the idle sweep.
